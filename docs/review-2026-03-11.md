## Phase 1 — Security & Correctness

Release-blocking. Two fully independent tracks.

---

### Track A — `src/security/target_policy.rs`

**A1 · HIGH · SSRF bypass via IPv4-mapped IPv6**

`::ffff:127.0.0.1` parses as `IpAddr::V6`; `v6.is_loopback()` returns `false` so the blocklist does not fire. On Linux dual-stack sockets, `TcpStream::connect` routes this to `127.0.0.1`. Fix: call `to_ipv4_mapped()` on any `IpAddr::V6` and apply the IPv4 checks when a mapped address is detected.

```
src/security/target_policy.rs:67-93
```

**A2 · MEDIUM · 6to4 and NAT64 IPv6 ranges not blocked**

`2002::/16` (6to4) and `64:ff9b::/96` (NAT64) encode arbitrary IPv4 addresses. `2002:7f00:0001::` encodes `127.0.0.1`. On networks with NAT64 relay support these reach internal hosts. Add explicit prefix checks for both ranges.

```
src/security/target_policy.rs:67-93
```

**A3 · MEDIUM · `fec0::/10` deprecated site-local space allowed**

The test at line 311-319 explicitly asserts these addresses are `is_ok()`. RFC 3879 deprecated this range; addresses in it can still reach internal hosts on some enterprise networks. Either block the range or add a documented comment explaining why it is intentionally permitted.

```
src/security/target_policy.rs:311-319
```

**A4 · LOW · `check_allowed` pub but only used by tests**

The function at line 16 is `pub` but only serves as a test convenience wrapper around `check_allowed_with_policy`. Mark `#[cfg(test)]` or remove it; test code can call `check_allowed_with_policy(ip, false)` directly.

```
src/security/target_policy.rs:16-18
```

### Track B — `src/routes.rs` + `src/main.rs`

**B1 · HIGH · `max_concurrent_handshakes` is validated but never enforced**

The config field is parsed, clamped, and exposed in `/api/meta`, but no `Arc<Semaphore>` is constructed. 7 ports × 10 IPs = 70 concurrent TLS connections can be opened per request regardless of the configured cap. Create the semaphore in `AppState` and acquire it in `inspect_port` before each handshake.

```
src/routes.rs:487-497
src/config.rs:53-54
src/state.rs (add semaphore field)
```

**B2 · HIGH · `max_ips_per_hostname` is validated but never enforced**

`do_inspect` never slices `allowed_ips` to the configured cap before the fan-out loop. The only IP-reduction path is the GCRA rate limiter, which is a separate mechanism. Add an explicit truncation of `allowed_ips` to `config.limits.max_ips_per_hostname` before inspection begins.

```
src/routes.rs:366-383
```

**B3 · MEDIUM · Split request-ID identity**

`request_id_middleware` generates a UUID v7 and writes it to the `X-Request-Id` response header (`main.rs:118`). `do_inspect` independently generates a second UUID v7 for the `request_id` JSON field (`routes.rs:338`). A client correlating header with body sees different values. Fix: read the ID from the request extension set by the middleware, or remove the middleware and generate once inside `do_inspect`.

```
src/main.rs:118-127
src/routes.rs:338
```

**B4 · MEDIUM · IP deduplication uses `O(n²)` `Vec::contains`**

`resolve_hostname` deduplicates IPs with a `Vec::contains` loop. Replace with a `HashSet` insertion idiom used elsewhere in the codebase.

```
src/routes.rs:832-839
```

**F1 · HIGH · Request ID not injected into tracing span**

`do_inspect` emits zero structured log entries. A production operator cannot tell from logs whether a given hostname was inspected, how many IPs were resolved, or how long the inspection took. Add:
- `tracing::info!(hostname, client_ip = %client_ip, request_id, "inspect start")` at entry
- `tracing::info!(hostname, ip_count, duration_ms, request_id, "inspect complete")` at exit

## Phase 7 — Low-Priority Hardening

Lower urgency; no strict ordering required between items.

---

**P7-1 · LOW · Rate limiter DashMap never evicts entries**

Every unique source IP and target hostname creates a permanent entry. Under sustained scanning with diverse IPs, heap grows without bound. Consider using a TTL-aware structure or periodic pruning of entries older than the rate window.

```
src/security/rate_limit.rs:34-38
```

**P7-2 · LOW · `RateLimitState::remaining_budget` overestimates available capacity**

`remaining_budget` returns the static burst size regardless of actual consumption. When the burst has been partially consumed, `ip_budget` in `do_inspect` overestimates the remaining budget, potentially causing a subsequent `check_cost` call to fail unexpectedly.

```
src/security/rate_limit.rs:64-66
```

**P7-3 · LOW · Metrics endpoint has no authentication**

If `metrics_bind` is changed to `0.0.0.0:9090`, request counters and rate-limit hit data become publicly accessible. Document the intended deployment model and add a note in the README that the metrics port must not be publicly exposed.

```
src/main.rs:207-229
README.md (add note)
```

**P7-4 · LOW · Single-label hostnames pass input validation**

`localhost`, `intranet`, `corphost` pass `validate_hostname`. They are only blocked post-resolution if the resolved IP is in a blocked range. Split-horizon DNS names that resolve to public-looking IPs would not be blocked. Consider rejecting single-label hostnames at input validation time.

```
src/input.rs:158-191
```

**P7-5 · LOW · `spawn_blocking` wrapping `rt.block_on` antipattern**

The `rand 0.9` `!Send` workaround at `routes.rs:457-474` occupies a blocking thread for async work, limiting concurrency under the blocking thread pool ceiling. Track whether the upstream `rand`/`mhost` issue is resolved; replace with a direct async call when possible.

```
src/routes.rs:457-474
```

**P7-6 · LOW · `docs/quality.md` contains stale references**

§11.2-11.3 references the removed `grade=true`/`grade=false` query parameter and a `tlsight_grade` localStorage toggle that was deleted. Remove or update these test case descriptions.

```
docs/quality.md:686-699
```

**P7-7 · LOW · `do_inspect` is a 360-line orchestration function**

All domain logic — IP resolution, rate limiting, enrichment, DNS, TLS, validation, quality, response building — lives in one function. Not a correctness issue today, but splits naturally into: `resolve_and_filter_ips`, `run_dns_checks`, `run_inspection_fan_out`, `assemble_response`. This refactor should follow Phase 5 Track O to avoid conflicts.

```
src/routes.rs:282-700
```
